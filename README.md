# Project Orion

`Project Orion` — это мультиагентный сервис генерации кода с `RU-first` подходом, рассчитанный на быстрое создание `MVP`-продуктов для `SMB`, фаундеров и небольших команд. Пользователь вводит задачу в интерфейсе, система запускает последовательный `SDLC workflow`, показывает прогресс работы агентов, считает токены и формирует итоговый набор артефактов: код, документы, историю выполнения и preview-результат.

## Что входит в проект

- `apps/api` — серверная часть на `NestJS`: `API`, `orchestrator`, `agent runtime`, `billing`, `token tracking`, `MCP registry`, история запусков и управление провайдерами.
- `apps/web` — клиентская часть на `Next.js`: `workspace`, чатовый ввод задачи, `SDLC timeline`, `token dashboard`, авторизация, профиль и история запусков.
- `packages/domain` — общий доменный слой: контракты, схемы, типы, конфигурации агентов, `workflow graph`, правила budget control и provider-модели.
- `packages/ui` — общий набор UI-компонентов, используемых на фронтенде.
- `infra/docker` — локальная инфраструктура для разработки.
- `infra/opentofu` — базовая `Infrastructure as Code`-заготовка для облачного развёртывания.
- `infra/helm` — `Helm chart` для деплоя в `Kubernetes`.
- `docs` — документация по архитектуре и развёртыванию.

## Что делает сервис

`Project Orion` принимает пользовательский prompt, создаёт запуск (`run`), строит фиксированную последовательность этапов `SDLC`, передаёт задачу агентам по очереди, контролирует расход токенов, сохраняет артефакты и формирует итоговый результат. Система делает акцент на прозрачности: пользователь видит, какой агент работает, сколько ресурсов уже потрачено, какие документы и результаты уже готовы, и почему выполнение могло быть остановлено.

## Архитектурный подход

- Используется `modular monolith`, а не `microservices`, чтобы быстрее вывести продукт в рабочее состояние и не усложнять эксплуатацию на этапе проверки гипотезы.
- Используется фиксированный `workflow graph`, а не свободное взаимодействие агентов, чтобы снизить риск каскадных ошибок и упростить контроль статусов, таймаутов и budget.
- Используются typed `artifacts` и typed `handoff payloads`, чтобы результаты агентов были структурированными и пригодными для передачи следующему этапу.
- Используется `provider adapter layer`, чтобы можно было подключать разные `LLM providers`, не привязываясь к одному вендору.
- `MCP integration` встроена как управляемый слой инструментов с allowlist-политиками и аудитом вызовов.

## Технологии проекта

### Backend

- `NestJS`
  Используется как основа серверного приложения. Отвечает за модульную структуру `API`, маршруты, `guards`, `dependency injection`, сервисы оркестрации и серверную бизнес-логику.

- `Prisma`
  Используется как `ORM` и слой доступа к данным. Через него описана схема базы данных, модели пользователей, запусков, шагов, артефактов, `token ledger`, `MCP servers` и журналов выполнения.

- `PostgreSQL`
  Используется как основная `relational database`. Хранит пользователей, сессии, баланс токенов, историю запусков, шаги workflow, артефакты и служебные данные системы.

- `Redis`
  Используется как `in-memory storage` для очередей, временного состояния, heartbeat-механик, блокировок и будущего `worker coordination`.

- `BullMQ`
  Используется как основа для `job processing` и фонового выполнения задач. Нужен для запуска и контроля долгих процессов оркестрации вне основного HTTP-запроса.

- `JWT`
  Используется для `authentication` и работы с `access token` / `refresh token`. Позволяет реализовать безопасную авторизацию и управление сессиями.

- `Argon2id`
  Используется для безопасного хеширования паролей. Нужен для защиты учётных данных пользователей.

- `Zod`
  Используется для схем, контрактов и валидации доменных структур. Помогает формализовать `run input`, `agent result`, usage-объекты и другие данные, которые передаются между слоями.

- `class-validator` и `class-transformer`
  Используются на уровне `NestJS DTO` для валидации входящих HTTP-запросов и преобразования данных.

### Frontend

- `Next.js`
  Используется для построения `web application` с `App Router`. Отвечает за страницы, интерфейс рабочего пространства, `dashboard`, профиль, авторизацию и статические/серверные UI-секции.

- `React`
  Используется как базовая библиотека пользовательского интерфейса. На нём собраны компоненты страниц, карточки, панели статусов и dashboard-элементы.

- `TypeScript`
  Используется во всём проекте как основной язык разработки. Обеспечивает типизацию, согласованность контрактов между frontend и backend и уменьшает число ошибок на этапе сборки.

- `Tailwind CSS`
  Используется для стилизации интерфейса. Позволяет быстро собирать консистентный UI, управлять layout, цветами, отступами и состояниями компонентов.

- `TanStack Query`
  Подготовлен как база для работы с `server state` на фронтенде. Нужен для получения данных по запускам, токенам, профилю, истории и кэширования ответов API.

### Shared packages

- `packages/domain`
  Это общий слой проекта, где собраны доменные типы, `schemas`, provider-конфигурации, правила budget control, конфигурации агентов и логика `workflow`.

- `packages/ui`
  Это общий UI-слой для переиспользуемых компонентов интерфейса, чтобы фронтенд не дублировал простые элементы представления.

### Инфраструктура и развёртывание

- `Docker`
  Используется для контейнеризации сервиса и локального запуска зависимостей. Нужен для повторяемой среды разработки и подготовки к облачному деплою.

- `Docker Compose`
  Используется для быстрого локального запуска `PostgreSQL`, `Redis` и `MinIO` как S3-compatible storage.

- `MinIO`
  Используется как локальная реализация `S3-compatible object storage`. Подходит для хранения артефактов, архивов и preview-результатов.

- `Helm`
  Используется для упаковки и управления деплоем приложения в `Kubernetes`. Позволяет параметризовать развёртывание web и api-частей.

- `OpenTofu`
  Используется как `Infrastructure as Code`-инструмент для подготовки облачной инфраструктуры и профилей развёртывания под российские облака.

### Observability и интеграции

- `MCP`
  Используется как слой подключения внешних инструментов и серверов. В проекте он нужен для контролируемого доступа агентов к инструментам, регистрации серверов, политик allowlist и аудита вызовов.

- `LLM provider adapter layer`
  Это внутренний адаптерный слой для подключения различных `LLM APIs`, включая `RU-first` сценарии. Он нормализует вызовы моделей, usage-метрики, токены, стоимость и возможности `JSON mode` / `tool calling`.

### Тестирование и качество

- `Vitest`
  Используется для `unit tests` и базовых integration-проверок. На нём проверяются workflow-правила, aggregation usage, budget accounting и базовые сценарии работы сервисов.

- `GitHub Actions`
  Используется для `CI pipeline`. Нужен для автоматического прогона тестов и сборки при изменениях в репозитории.

## Почему выбран именно такой стек

- `TypeScript` выбран, чтобы frontend, backend и shared domain-слой говорили на одном языке и использовали одни и те же контракты.
- `NestJS` выбран, потому что он хорошо подходит для модульного backend с `auth`, `orchestrator`, `billing`, `history`, `MCP` и другими бизнес-модулями.
- `Next.js` выбран, потому что позволяет быстро создать современный интерфейс продукта с хорошей структурой страниц и удобной интеграцией с API.
- `PostgreSQL` выбран, потому что в проекте много связанной бизнес-сущностной логики, историй, запусков, шагов и журналов, которые надёжно укладываются в relational model.
- `Redis` выбран, потому что нужен быстрый слой для фоновых задач и краткоживущего состояния.
- `Prisma` выбран, потому что он ускоряет работу с БД, даёт понятную схему и уменьшает объём шаблонного кода.
- `Docker` и `Helm` выбраны, потому что проект сразу ориентирован на облачное развёртывание и должен быть переносимым между разными окружениями.

## Быстрый старт

1. Скопируйте `.env.example` в `.env`.
2. Запустите инфраструктуру через `docker compose up -d`.
3. Установите зависимости через `pnpm install`.
4. Сгенерируйте клиент базы данных через `Prisma`.
5. Запустите `apps/api`.
6. Запустите `apps/web`.
